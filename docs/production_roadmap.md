# Production Roadmap

デザインMVPの次は、EDINET実データ、出典管理、分析アルゴリズム、公開運用の順で固めます。

## Phase 1: いったん公開できる状態

目的:

- GitHubから安全にデプロイできる
- APIキーをSecretsで管理できる
- サンプルCSVだけでもアプリが落ちない
- EDINET検索は任意機能として動く

作業:

- `Dockerfile` と `.dockerignore` を維持する
- Streamlit CloudまたはRenderへデプロイする
- `EDINET_API_KEY` をSecrets/環境変数で設定する
- 生成物、DB、EDINETファイルをGitHubに含めない
- 公開URLで基本操作を確認する

## Phase 2: EDINET API取得の本格化

目的:

- 証券コードから対象企業の有価証券報告書を安定して探せる
- 取得履歴と出典を残せる
- API失敗、欠損、検索期間不足をユーザーに説明できる

作業候補:

- 取得ジョブID、取得日時、検索期間、検索条件をSQLiteへ保存
- EDINET `docID` ごとの取得状態を管理
- API失敗時のリトライとユーザー向けエラーメッセージを改善
- 証券コード、EDINETコード、会社名の対応を永続化
- CSV ZIP、XBRL、PDFの保存先を `output/raw_filings/` から将来R2/S3へ移行
- 取得済みファイルを再利用し、同じdocIDを何度も取りに行かない

## Phase 3: XBRL/CSV解析

目的:

- EDINETから取得したファイルを、現行の分析用データ形式へ変換する
- サンプルCSVから実データへ段階的に切り替える

作業候補:

- EDINET CSV ZIPを展開し、財務諸表CSVを識別
- 売上高、営業利益、当期純利益、総資産、純資産、負債、営業CF、投資CFなどのタグ候補をマッピング
- 企業ごと、年度ごとの正規化テーブルを作る
- 欠損値は落とさず `推定不可` として理由を保存
- タグ揺れ、連結/単体、IFRS/JGAAPの違いを扱う

## Phase 4: 分析アルゴリズム高度化

目的:

- 表とグラフだけではなく、差が出た理由を説明できる
- 課題の必須部分と＋αを明確に分ける
- 汎用分析でも業種ごとの論点を使える

強化対象:

- ROA/ROE分解
- 売上増減分析
- 利益増減分析
- 損益分岐点分析
- 安全余裕率
- 営業レバレッジ
- FCF分析
- PER/PBR比較
- 4P/4C
- 顧客関係
- バリューチェーン
- 外部環境シナリオ

業種別観点:

- カフェ・外食: 店舗数、客数、客単価、原材料費、人件費、家賃、FC比率
- 航空: 燃油費、為替、旅客単価、搭乗率、機材投資、国際線/国内線、外部環境
- 小売: 既存店売上、粗利率、在庫回転、物流、出店/退店
- 卸売: 仕入、物流、在庫、価格転嫁、取引先集中

## Phase 5: LLM連携

目的:

- 数値計算はアプリ側で確定する
- LLMには文章の磨き込み、論点漏れチェック、改善提案を依頼する
- 投資助言表現を避ける制約を維持する

方式:

- アプリがWordレポートを生成
- 同時にLLM用Markdownプロンプトを生成
- ChatGPT Proなどに貼り付けてレビューを受ける
- 将来はOpenAI API連携を検討するが、APIキー管理、費用、ログ保存、個人情報に注意する

LLMに任せないもの:

- EDINETから取得した数値の確定
- 財務指標の計算式
- 課題条件の合否判定
- 投資判断や売買推奨

LLMに任せるもの:

- 文章の自然さ
- 論理のつながり
- 追加分析の提案
- 章立て改善
- 欠損データ注記の読みやすさ

## Phase 6: Webアプリ本格化

目的:

- 誰でもブラウザから使える
- 長時間取得やレポート生成を安全に扱える
- データ保存と再利用ができる

将来構成:

```text
Frontend: Cloudflare Pages / React / Next.js
Backend: FastAPI
Worker: EDINET取得・XBRL解析・Word生成
Database: PostgreSQL
Storage: Cloudflare R2 / S3
Auth: 必要になった段階で追加
```

Streamlitは、当面のMVP・検証・管理画面として残せます。
